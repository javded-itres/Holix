# Стратегия лицензирования Holix

Документ описывает варианты изменения лицензирования агента Holix с учётом целей:

- агент остаётся **open source**;
- на его основе планируются **бизнес-приложения**, **корпоративные обёртки** и **SaaS**;
- нужна защита коммерческой модели без потери доверия сообщества.

**Текущее состояние (2026):** ядро Holix распространяется под **MIT License** (`LICENSE`, `pyproject.toml`).

---

## Почему MIT — не финальный ответ

MIT максимально свободна: open source и коммерческое использование разрешены **всем**, включая конкурентов.

| Плюс MIT | Минус MIT |
|----------|-----------|
| Доверие сообщества, простой вход для корпораций | Любой может форкнуть ядро и запустить идентичный hosted-сервис |
| Нет трений с PyPI / GitHub | Слабая защита бизнес-модели «Holix как SaaS» |
| Удобно для встраивания в продукты | Нет обязательства отдавать улучшения обратно |

Для цели «свой SaaS + корпоративные обёртки + открытый агент» MIT удобна на старте, но **не защищает** от бесплатного клона облачного сервиса.

---

## Вариант 1. Оставить MIT + монетизировать вокруг кода

**Схема:** ядро Holix остаётся MIT; доход — из сервиса и enterprise-слоя, а не из закрытия агента.

| Open source (MIT) | Коммерческое |
|-------------------|--------------|
| CLI, память, skills, MCP, локальный агент | Managed SaaS (например holix-agent.ru) |
| Gateway API (базовый) | Хостинг, SLA, поддержка |
| Документация, примеры | Enterprise onboarding, кастомизация |

**Плюсы:** доверие сообщества, лёгкий вход для корпораций.  
**Минусы:** конкурент может поднять «Holix Cloud» на том же коде без отдачи сообществу.

Подходит, если готовы конкурировать **брендом, скоростью, экосистемой и enterprise-слоем**, а не лицензией.

---

## Вариант 2. Open Core (рекомендуемая основная стратегия)

**Схема:** открытое ядро + отдельные коммерческие модули для бизнеса.

```
holix (MIT/Apache)           — агент, runtime, память, tools
holix-gateway (MIT)          — базовый API, single-tenant
holix-enterprise (commercial)— multi-tenant, billing, SSO, audit, RBAC
holix-cloud (SaaS)           — хостинг и панель IT-RES
```

У проекта уже есть задел: `/api/holix/`, профили, gateway, SaaS-примеры в документации — это естественная граница «community vs commercial».

### Что оставить open source

- runtime агента (LangGraph, tools, memory);
- CLI / TUI;
- локальный gateway для одного инстанса;
- интеграции (Telegram, MCP, skills).

### Что вынести в commercial / Enterprise

- multi-tenant isolation (организации, квоты, биллинг);
- SSO (SAML/OIDC), RBAC, audit log;
- централизованный admin для N клиентов;
- compliance (152-ФЗ, хранение логов, DLP);
- managed updates, HA, backup.

**Плюсы:** честный open source + понятная монетизация; корпорации часто покупают не код, а **поддержку и compliance**.  
**Минусы:** нужна чёткая граница репозиториев; нельзя «утягивать» community-фичи в enterprise без обратной отдачи.

## Вариант 3. Dual License: AGPL (community) + Commercial License (B2B)

**Схема:** публично — **AGPL-3.0**; для корпораций, которые не хотят открывать код — **коммерческая лицензия** от правообладателя.

| Сценарий | Что происходит |
|----------|----------------|
| Разработчик ставит локально | Бесплатно, AGPL |
| Компания встраивает во внутренний продукт (не отдаёт наружу) | Обычно допустимо под AGPL |
| Кто-то поднимает **SaaS на базе Holix** | Должен открыть модификации **или** купить commercial license |
| Собственный SaaS IT-RES | Нужны права на весь код (CLA, единый copyright holder) |

Модель MariaDB, Qt (исторически), многих B2B open source проектов.

**Плюсы:** сильная защита от «бесплатного клона SaaS».  
**Минусы:** AGPL пугает часть enterprise (юристы); нужен **CLA** от контрибьюторов; смена MIT → AGPL — отдельный юридический шаг.

---

## Вариант 4. Apache 2.0 вместо MIT (при Open Core, без AGPL)

Если остаётесь на permissive-лицензии, но хотите лучше для корпораций:

- явная **patent grant** (защита от патентных исков);
- привычнее для enterprise procurement, чем MIT.

Для защиты SaaS Apache **не лучше** MIT — оба позволяют hosted-конкурентам то же самое.

---

## Вариант 5. BSL / SSPL — не рекомендуется как первый шаг

- **BSL** (HashiCorp): через N лет становится open — community часто воспринимает как «псевдо-open».
- **SSPL** (MongoDB): очень агрессивна к SaaS; многие cloud-провайдеры и юристы её избегают.

Имеет смысл только при сознательной жертве community-ростом ради жёсткой защиты hosted-модели.

---

## Рекомендация для Holix

Учитывая планы: обёртки, корпоративные агенты, SaaS + open source.

### Оптимальная комбинация

1. **Сейчас:** не спешить менять MIT на всём репозитории.
2. **Через 1–2 релиза:** формализовать **Open Core**:
   - `Helix` (agent core) — **Apache 2.0** или MIT;
   - `holix-enterprise` / `holix-cloud` — отдельный репозиторий, **commercial**.
3. **Параллельно:** trademark policy (бренд «Holix»; форки без confusing similarity).
4. **Перед dual license:** ввести **CLA** (Contributor License Agreement), чтобы лицензировать код и как OSS, и как commercial.

### Пример границы лицензий

| Компонент | Лицензия |
|-----------|----------|
| Agent runtime, memory, tools | Apache 2.0 |
| CLI, TUI, doctor | Apache 2.0 |
| Gateway (single profile, self-hosted) | Apache 2.0 |
| Multi-tenant control plane | Commercial |
| Billing, orgs, SSO, audit | Commercial |
| holix-docs (сайт) | MIT/Apache (маркетинг) |
| Managed cloud IT-RES | Сервис + SLA (не лицензия на код) |

---

## Юридический минимум

1. **LICENSE** в каждом репозитории + `NOTICE` для Apache.
2. **CLA** — до активных внешних контрибьюторов.
3. **Trademark Guidelines** — имя Holix, логотип, «official» vs fork.
4. **Commercial License FAQ** — кто и за что платит (SaaS operator, OEM, on-prem enterprise).
5. **Third-party licenses** — совместимость зависимостей (aiogram, LangGraph, chromadb и т.д.) с выбранной моделью, особенно при переходе на AGPL.

## Сравнение моделей

| Модель | Open source | Свой SaaS | Защита от клона | Сложность |
|--------|-------------|-----------|----------------|-----------|
| MIT (как сейчас) | ✅ | ✅ | ❌ | низкая |
| Open Core | ✅ ядро | ✅ | ⚠️ средняя | средняя |
| Dual AGPL + Commercial | ✅ | ✅* | ✅ | высокая |
| BSL / SSPL | ⚠️ | ✅ | ✅✅ | высокая + риск community |

\* при едином copyright и CLA.

---

## Следующие шаги (черновики)

1. **Open Core roadmap** — какие модули в open repo, какие в commercial.
2. **Commercial License** — три тарифа: self-hosted enterprise, OEM, SaaS operator.
3. **CLA** + обновление `LICENSE` и структуры репозиториев.

### Выбор приоритета

| Приоритет | Рекомендация |
|-----------|--------------|
| Максимальный рост community | MIT/Apache + Open Core + сильный бренд и SaaS |
| Максимальная защита от SaaS-конкурентов | Dual AGPL + Commercial License |
